Background Images in Twine

Three methods for setting passage backgrounds, and when to use each one

Adding a background image to a Twine passage is one of the most effective ways to establish place, mood, and atmosphere without adding image elements to every passage individually. This guide covers all three workable methods, explains why local images cause problems, and gives you the simplest path to a working background for your project.

→

The three approaches

There are three ways to add a background image in Harlowe. They differ in how images are stored, how much setup is required, and whether the background changes between passages.

Method 1
(enchant:) macro + hosted URL

Place an (enchant:?page, (bg:"url")) macro in a header-tagged passage. The image is loaded from a hosted URL each time the story runs.

Simplest code No conversion Needs internet
Method 2
CSS passage tags + hosted URL

Tag individual passages and write CSS rules in the Story Stylesheet targeting each tag. Different passages get different backgrounds automatically.

Per-passage control No conversion Needs internet
Method 3
Base64-encoded image in CSS

Convert the image to a Base64 string and embed it directly in the story's CSS Stylesheet. The story is fully self-contained — no internet connection needed.

Works offline Self-contained Large file size Unwieldy to edit
Why not local files?
The local file problem

If you write background-image: url("photo.jpg") in CSS, browsers block the load when the .html is opened directly from your computer — a security restriction called the same-origin policy. The (bg:) macro sidesteps this, but CSS url() does not.

Breaks in CSS url() Works via (bg:) macro

Method 1 — The (enchant:) Macro

This is the simplest method. You write a small block of Harlowe macros in a special passage that runs on every page, and the image appears as the background for the whole story.

Setting up the header passage

  1. In Twine, create a new passage. Name it anything — Style is conventional.

  2. Open the passage, click the tag icon, and add the tag header. This tells Harlowe to run this passage's content before every other passage.

  3. Paste the following code into the passage, replacing the URL with your own image: 

    {
(enchant: ?page, (bg:"https://your-image-url.jpg"))
(enchant: ?passage, (text-colour:black) + (bg:#ffffff) + (css:"padding:1.5em"))
(enchant: ?link, (colour:navy))
}

  4. Click Play to test. The background should appear behind the white text box on every passage.

💡 What those macros do

(enchant: ?page, (bg:"url")) — sets the full-page background to your image.

(enchant: ?passage, ...) — styles the text box: white background, black text, padding. Without this, text sits directly on the image and is hard to read.

(enchant: ?link, (colour:navy)) — makes links navy blue so they stand out against the white text box. Change this to any colour you like.

The { } curly braces collapse whitespace so the macros produce no visible output.

Using different backgrounds per passage with Method 1

Method 1 applies the same image everywhere. If you want different backgrounds on different passages, you have two options:

Option A: Add (enchant: ?page, (bg:"different-url.jpg")) directly at the top of individual passages where you want an override. This replaces the header's background just for that passage.

Option B: Switch to Method 2, which is cleaner when you have several different backgrounds.

⚠️ Important

The image URL must be a full https:// address — not a filename like "forest.jpg". Local filenames work during Twine's preview mode (Play/Test) but break in the exported .html file unless the image is in the same folder. Use a hosted URL or Base64 (Method 3) for a reliably portable file.

2

Method 2 — CSS Passage Tags

This method lets each passage have its own background image. You tag individual passages in Twine, then write CSS rules in the Story Stylesheet that target those tags. Harlowe automatically adds each passage's tags to the tw-story element when that passage is displayed, so the right CSS rule fires at the right time.

Step 1 — Add rules to the story's CSS Stylesheet

  1. In your Twine story, click Story in the top navigation bar, then click Stylesheet.

  2. Add one rule per background image. Each rule uses a tw-story[tags~="tagname"] selector: 

    /* Default background for all passages */
tw-story {
  background-image: url('https://your-default-bg.jpg');
  background-size: cover;
  background-position: center;
  background-attachment: fixed;
}

/* Forest passages — tag: forest */
tw-story[tags~="forest"] {
  background-image: url('https://your-forest-image.jpg');
  background-size: cover;
  background-position: center;
}

/* City passages — tag: city */
tw-story[tags~="city"] {
  background-image: url('https://your-city-image.jpg');
  background-size: cover;
  background-position: center;
}

/* Style the text box so it's readable over any background */
tw-passage {
  background: rgba(255, 255, 255, 0.92);
  padding: 1.5em 2em;
  border-radius: 4px;
  max-width: 700px;
  margin: 2em auto;
}

  3. Close the stylesheet editor.

ℹ️ Why background-attachment: fixed?

Adding background-attachment: fixed to the default rule stops the background image from scrolling with the page — it stays locked in place while the text box scrolls over it. This looks more polished but can cause visual glitches on iOS Safari. Leave it out if you are targeting mobile.

Step 2 — Tag your passages

  1. In the story map, double-click a passage to open it in the editor.

  2. Click the tag icon (looks like a label) in the passage editor toolbar.

  3. Type your tag name — for example forest — and press Enter.

  4. The tag must match exactly what you wrote in the CSS: same spelling, same capitalisation. Forest and forest are different.

  5. Repeat for every passage that needs a background. Passages with no tag use the default tw-story rule (or no image if you did not write one).

💡 Tip — adding multiple tags

A passage can have more than one tag. You can combine a background tag (like forest) with a mood tag (like tense) and write separate CSS rules for each. Both sets of styles will apply at the same time.

3

Method 3 — Base64-Encoded Images

Base64 encoding converts a binary image file into a long string of text characters that can be embedded directly in HTML or CSS. The result is a completely self-contained .html file that displays the image correctly whether it is online or offline, without needing any companion files.

The downside is size: a 500KB image becomes roughly 667KB of text, and even a single large image makes the stylesheet unwieldy to look at. Use this method when portability matters more than file tidiness — for a class submission, for example.

Step 1 — Convert your image to Base64

  1. Go to base64-image.de in your browser.

  2. Click Upload image and select your file, or paste an image URL.

  3. Once converted, click Copy CSS. This copies a ready-to-use CSS declaration like: 

    background-image: url('data:image/jpeg;base64,/9j/4AAQSkZJRgAB...');
/* ... thousands of characters ... */

  4. Open your Twine story, select Story from the top navigation bar, then select Stylesheet.

  5. Write your CSS rule and paste the copied declaration inside it: 

    tw-story {
  background-image: url('data:image/jpeg;base64,/9j/4AAQ...');
  background-size: cover;
  background-position: center;
}

tw-passage {
  background: rgba(255, 255, 255, 0.92);
  padding: 1.5em 2em;
  max-width: 700px;
  margin: 2em auto;
}
⚠️ Image size matters

Keep images under 300–400KB before encoding. A 3MB photograph will generate a ~4MB Base64 string, making the entire stylesheet slow to load and difficult to edit. Resize or compress your image first — tools like squoosh.app (free, browser-based) can reduce file size dramatically with minimal quality loss.

Using Base64 with passage tags (Method 2 + Method 3 combined)

You can combine Base64 encoding with passage tags. Just use the same tw-story[tags~="tagname"] selector, but use the Base64 data URL instead of an https:// URL. This gives you per-passage backgrounds with full offline portability — at the cost of a very large stylesheet.

+

Styling the Text Box

Whatever method you use for the background, you need to make sure the text is readable over it. The default Harlowe text box has no background of its own and sits directly on top of whatever is behind it. These are the most useful CSS properties for the passage text box: 

tw-passage {
  /* Semi-transparent white box — adjust opacity as needed */
  background: rgba(255, 255, 255, 0.90);

  /* Or a fully opaque dark box for dramatic effect */
  /* background: rgba(0, 0, 0, 0.75); color: white; */

  /* Space inside the box */
  padding: 1.5em 2em;

  /* Limit width so text doesn't stretch too wide */
  max-width: 680px;

  /* Centre the box on the page */
  margin: 2em auto;

  /* Optionally round the corners */
  border-radius: 6px;
}
💡 Design principle

The passage box and the background are in tension: a vivid background image wants to be seen, but readable text needs contrast. A semi-transparent white box (rgba(255,255,255,0.88)) is usually the right balance — it lets a sliver of the image show through the edges while keeping text crisp. Adjust the last value (opacity) between 0.7 and 1.0 to taste.

Setting background size and position

These two CSS properties control how the image fits the page:

Property & value Effect Best for
background-size: cover Scales the image to fill the entire screen, cropping if necessary. No empty areas. Photographs and full-bleed scenes
background-size: contain Scales the image to fit entirely within the screen. May leave empty bars at the sides or top. Illustrated maps or images where nothing should be cropped
background-position: center Centres the image in both axes when cropping occurs. Most images
background-position: top center Aligns the top of the image with the top of the screen. Images where the important content is near the top
background-repeat: no-repeat Prevents tiling. Usually needed with contain. Any non-tiling image

Hosting Your Images

Methods 1 and 2 require a hosted image URL — an https:// address that loads the image from the internet. Here are the most practical options for students:

Option How to get a direct URL Notes
Flickr Open the photo page → click the download arrow → choose a size → right-click the opened image → Copy image address. URLs are stable. Free account. Images must be set to public.
Unsplash / Pexels Open any photo → right-click → Copy image address. Or use the direct download URL. Free to use. Large library of high-quality images.
Wikimedia Commons Open the file page → click the image to get the full-resolution view → copy the URL from the address bar. Stable long-term URLs. Check the licence before use.
GitHub Upload image to a public repository → open the raw file → copy the raw URL (starts with raw.githubusercontent.com). Reliable. Requires a free GitHub account.
Same folder as .html Use a relative filename like "images/forest.jpg". Works when the exported .html and images folder are kept together. Breaks if separated.
⚠️ Check your URL format

A working image URL ends with a file extension like .jpg, .png, or .webp. URLs that end in a page address (like flickr.com/photos/user/12345) are webpage URLs — they point to an HTML page that contains an image, not to the image itself. Always open the image in its own tab to get the direct URL.

Troubleshooting

Problem Most likely cause and fix
Background shows in Twine preview but not in the exported file You used a local filename ("forest.jpg") in CSS url(). Browsers block local files in CSS. Either host the image and use a full https:// URL, or switch to Base64 (Method 3), or keep the image in the same folder as the .html file and use a relative path.
Background does not appear at all Check the URL is correct: open it in a browser tab — you should see only the image, not a webpage. Also check the CSS selector is targeting the right element. In Harlowe, use tw-story, not body or html.
CSS tag rule is not triggering on a passage The tag in Twine and the tag in the CSS do not match exactly. Check for differences in capitalisation, spaces, or punctuation. Forest and forest are different tags.
(enchant:?page, (bg:"url")) fails silently — no image, no error The URL is either wrong or the image host is blocking cross-origin loads. Test the URL in a browser tab first. If the image loads there but not in Twine, the host may not allow embedding — choose a different host.
Text is unreadable over the background image The tw-passage background has not been set. Add tw-passage { background: rgba(255,255,255,0.9); padding: 1.5em 2em; } to the story's Stylesheet, or add (enchant: ?passage, (bg:#ffffff) + (css:"padding:1.5em")) to the header passage.
Background image is stretched or pixelated Add background-size: cover; background-position: center; to the CSS rule. Without these, the browser tiles or stretches the image at its natural size.
Base64 string causes an error or the stylesheet won't save The string may be incomplete — Base64 strings are very long and easy to accidentally truncate when copying. Try the conversion tool again and use its dedicated copy button rather than selecting the text manually.
